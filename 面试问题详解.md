## 1. 简单描述项目，实现了怎样的功能，用了哪些技术栈

本项目是基于 **C++11** 实现的一个 RPC 分布式网络通信框架，使用 **CMake** 在 Linux 平台上构建。它将单体架构的本地方法调用重构为基于 TCP 网络通信的 RPC 远程方法调用，实现了同一台机器不同进程之间、不同机器之间的服务调用，适用于将单体架构拆分为分布式微服务部署。

**技术栈：**

| 组件 | 技术选型 | 作用 |
|---|---|---|
| 网络层 | Muduo | Reactor 高并发网络模型 |
| 序列化 | Protobuf | 二进制序列化、跨语言、代码生成 |
| 服务注册与发现 | ZooKeeper | 分布式协调，Watcher 实时感知 |
| 日志 | Glog | 异步、线程安全、自动分割 |
| 构建 | CMake | 跨平台构建 |

**核心分层：**

```
业务层：  UserService::Login()          stub.Login()
            ↑                              ↓
框架层：  KrpcProvider                 KrpcChannel
            ↑                              ↓
网络层：  Muduo TcpServer              raw TCP socket
            ↑                              ↓
注册中心： ZkClient ←──── ZooKeeper ────→ ZkClient
```

## 2. 为什么要使用 ZooKeeper 作为服务注册与发现中心？

> "ZooKeeper 是一个分布式协调工具，特别适用于分布式环境中的服务注册与发现。它的优点是高可用性和强一致性，通过 Watcher 机制可以实时感知服务的上线和下线。使用 ZAB 协议保证数据的一致性，确保每个节点的数据更新都是线性化的。缺点是写操作吞吐量不高，在大规模集群中可能成为瓶颈。"

### ZooKeeper 上的节点结构

```
/Kuser.UserServiceRpc              ← 永久节点（服务名）
  ├── /Login                       ← 永久节点（方法名）
  │   ├── /127.0.0.1:8000          ← 临时节点，数据存 "127.0.0.1:8000"
  │   └── /127.0.0.1:8001          ← 临时节点
  └── /Register                    ← 永久节点
      └── /127.0.0.1:8000          ← 临时节点
```

### 为什么方法节点是永久的，实例节点是临时的？

- **服务名 + 方法名节点是永久的**：即使所有实例下线，方法和路径还在，新实例上线直接添加即可
- **实例节点 `ip:port` 是临时的**（`ZOO_EPHEMERAL`）：服务端断开连接后，ZooKeeper 自动删除该节点。不需要服务端主动注销

### 客户端怎么发现服务？

代码实现在 `KrpcChannel::RefreshProviders()`：
1. `GetChildrenWithWatcher("/服务名/方法名")` 并注册子节点变化监听 → 拿到 `["127.0.0.1:8000", "127.0.0.1:8001"]`
2. 逐个 `GetData("/服务名/方法名/127.0.0.1:8000")` → 拿到 `"127.0.0.1:8000"`
3. 解析出 ip 和 port，存入 `m_providers` 列表

### 为什么节点名叫 `ip:port`，节点数据也存 `ip:port`？

ZooKeeper C API 的 `zoo_wget_children` 只返回子节点**名称**，不返回数据。虽然可以直接从节点名解析 ip:port，但为了通用性（节点名不一定是 ip:port 格式），代码选择再调一次 `GetData` 取值。

## 3. 消息传输为什么用 Protobuf，而不用 JSON？

| 协议 | 特点 |
|---|---|
| **JSON** | 明文 Key-Value，使用方便，但序列化后空间开销大，性能不行 |
| **Hessian** | 动态类型、二进制、紧凑，比 JSON 紧凑高效 |
| **Protobuf** | 二进制序列化，体积最小，速度最快，跨平台兼容，消息格式可升级 |

**Protobuf 在本项目中的额外优势：**
- **反射**：`ServiceDescriptor` / `MethodDescriptor` 让服务端可以动态创建请求/响应对象，框架代码不依赖具体服务类型
- **代码生成**：自动生成 Service / Stub 代码，业务方只需重写虚函数

## 4. 实现 RPC 时遇到了哪些难点？如何解决的？

### 4.1 TCP 粘包/拆包

TCP 是流式协议，没有消息边界。通过**自定义通信协议**解决：

```
请求包：[4B TotalLen(net)] + [4B HeaderLen(net)] + [Protobuf RpcHeader] + [Protobuf Args]
响应包：[4B TotalLen(net)] + [Protobuf ResponseData]
```

**服务端**用 while 循环 + peek 预读（`KrpcProvider::OnMessage`）：

```cpp
while (buffer->readableBytes() >= 4) {              // 至少能读 4 字节
    memcpy(&total_len, buffer->peek(), 4);           // peek 预读，不消费
    total_len = ntohl(total_len);
    if (buffer->readableBytes() < 4 + total_len) {   // 包没收全
        break;                                       // 等下次 OnMessage
    }
    buffer->retrieve(4);                              // 消费 total_len
    // ... 读 header_len → header → args → 分发业务
}
```

**客户端**用 `recv_exact` 循环接收，不收够不返回：

```cpp
ssize_t KrpcChannel::recv_exact(int fd, char* buf, size_t size) {
    size_t total_read = 0;
    while (total_read < size) {
        ssize_t ret = recv(fd, buf + total_read, size - total_read, 0);
        if (ret == 0) return 0;
        if (ret == -1) {
            if (errno == EINTR) continue;  // 信号中断，继续读
            return -1;
        }
        total_read += ret;
    }
    return total_read;
}
```

### 4.2 ZooKeeper 的学习与使用

- Watcher 机制：`GetChildrenWithWatcher()` 注册监听，节点变化时回调 `ChildWatcher` 设置脏标记
- **双重刷新**：Watcher 推送 + 10 秒定时兜底，防止 watcher 丢失
- 将 ZK 作为注册中心，实现服务动态发现

### 4.3 Protobuf 反射与序列化

利用反射避免硬编码具体类型：

```cpp
// 通用代码，不依赖具体请求/响应类型
Message* request  = service->GetRequestPrototype(method).New();
Message* response = service->GetResponsePrototype(method).New();
request->ParseFromString(args_str);
```

### 4.4 高并发网络处理

Muduo `setThreadNum(4)`，主线程 accept，4 个 IO 线程处理已建立连接的读写，网络 IO 与业务处理通过回调解耦。

## 5. 如何处理分布式系统中的故障？采取了哪些措施确保高可用性？

### 5.1 服务注册与发现

通过 ZooKeeper 实现动态注册与发现。Watcher + 10 秒定时双重刷新，服务宕机后客户端自动获取新节点。

### 5.2 超时重试

`CallMethod` 有 2 次重试机制：

```cpp
for (int attempt = 0; attempt < 2; ++attempt) {
    // 从连接池获取 fd → 找不到就 ConnectToAvailableProvider
    // 发送 → 收响应 → 反序列化
    // 任何一步失败 → MarkProviderFailed → RemoveFromPool → continue
}
```

### 5.3 故障隔离

3 秒内不重试已失败节点。代码实现在 `ProviderAlive()`：

```cpp
bool KrpcChannel::ProviderAlive(const ProviderInfo &provider) const {
    if (provider.alive) return true;
    auto now = std::chrono::steady_clock::now();
    return now - provider.last_failed >= std::chrono::seconds(3);
}
```

**为什么 3 秒自动恢复？** 可能是临时网络抖动；如果永久挂了，ZK 临时节点会自动删除，下次刷新就拉不到了。

### 5.4 负载均衡

在所有存活节点中**随机选一个**（`SelectProvider` 使用 `std::mt19937` + `uniform_int_distribution`）：

```cpp
std::vector<size_t> alive_indices;
for (size_t i = 0; i < m_providers.size(); ++i) {
    if (ProviderAlive(m_providers[i])) alive_indices.push_back(i);
}
size_t chosen_index = alive_indices[dist(gen)];
```

配合连接池策略：多实例场景调用成功后，从池中移除当前连接的 fd，迫使下次调用重新随机选节点。

## 7. RPC 是什么，它是如何工作的？

> "RPC 是一种通过网络从远程计算机请求服务的协议。核心思想是：客户端通过代理调用服务端方法，像调用本地方法一样，透明地隐藏网络通信细节。"

**CallMethod 完整流程（基于最新代码）：**

```
stub.Login(&controller, &request, &response, nullptr)
  │
  ├─ UserServiceRpc_Stub::Login()
  │     └─ channel_->CallMethod(descriptor()->method(0), ...)
  │
  └─ KrpcChannel::CallMethod()
        ├─ 1. EnsureZkClient() — 懒初始化 ZK 客户端
        ├─ 2. 从连接池取 fd：GetPooledFd(m_ip, m_port)
        ├─ 3. 池里没有 → ConnectToAvailableProvider()
        │      ├─ 需要刷新？→ RefreshProviders() 从 ZK 拉服务列表
        │      ├─ SelectProvider() 随机选存活节点
        │      ├─ 池里有该节点 fd → 复用
        │      └─ 没有 → newConnect() → AddToPool()
        ├─ 4. 序列化 request → args_str
        ├─ 5. 构建 RpcHeader（service_name, method_name, args_size）
        ├─ 6. 打包：[4B total_len][4B header_len][header][args]

        ├─ 9. recv_exact(fd, buf, response_len)
```

## 8. 如何看待微服务架构与传统单体应用架构的区别？

微服务架构通过将一个大型应用拆分为多个独立、自治的服务，提升系统的灵活性、可扩展性和可维护性。

**优势：**
- **独立部署与扩展**：每个服务独立部署，单服务故障不影响整个系统
- **高可用性**：服务间低耦合，故障可快速隔离
- **技术栈灵活**：不同服务可用不同技术栈

**挑战：**
- 服务间通信复杂度增加
- 分布式事务处理
- 数据一致性问题

## 9. TCP 粘包与拆包问题（详细）

### 成因

**粘包：**
- TCP 是流式协议，数据按字节流传输，没有消息边界
- 应用层发送频繁或来不及接收 → 多个小包被 TCP 缓冲区合并
- Nagle 算法将小包合并后再发送

**拆包：**
- 消息过大，超过 TCP 缓冲区大小，分多个包传输
- 网络拥塞或延迟

### 解决方案 — 自定义协议

```
请求：[4B TotalLen(net)] + [4B HeaderLen(net)] + [Protobuf RpcHeader] + [Protobuf Args]
响应：[4B TotalLen(net)] + [Protobuf ResponseData]
```

**为什么用长度前置而不是分隔符？**
- 二进制协议中分隔符可能出现在数据中，不可靠
- 长度前置是二进制协议的标准做法（HTTP Content-Length、gRPC 等）
- 仅额外 4 字节开销

**实现要点：**

服务端用 `peek` + while 循环解包（见 `KrpcProvider::OnMessage`），客户端用 `recv_exact` 精确读取字节（见 `KrpcChannel::recv_exact`）。所有整数字段使用 `htonl`/`ntohl` 网络字节序。

## 10. ZooKeeper 的 Watcher 机制

### 原理

Watcher 是一个**一次性观察者**。客户端注册对某节点的 Watch，当节点变化时，ZK 服务端通知客户端。

**特点：**
- 通知是**一次性**的（触发后失效）
- 通知**不返回节点数据**，只告知"变了"
- 客户端需自行根据事件类型做后续处理

### 本项目中的应用

```cpp
// 注册 Watcher — ZooKeeper 节点变化时回调 ChildWatcher
static void KrpcChannel::ChildWatcher(zhandle_t *zh, int type, int state,
                                       const char *path, void *watcherCtx) {
    if (type == ZOO_CHILD_EVENT) {
        KrpcChannel *channel = static_cast<KrpcChannel*>(watcherCtx);
        channel->m_providers_dirty = true;  // 设置脏标记，下次 CallMethod 触发刷新
    }
}

// RefreshProviders 中注册
children = zkclient->GetChildrenWithWatcher(method_path.c_str(),
    &KrpcChannel::ChildWatcher, this);
```

**双重刷新机制（代码在 `ConnectToAvailableProvider`）：**

```cpp
// 条件：watcher 触发脏标记 || 列表为空 || 超过 10 秒未刷新
if (m_providers_dirty || m_providers.empty() ||
    now - m_last_refresh_time > std::chrono::seconds(10)) {
    RefreshProviders(zkclient, service_name, method_name);
}
```

1. **Watcher 推送**：节点变更 → 回调设置 `m_providers_dirty = true`
2. **定时兜底**：每 10 秒强制刷新，防止 watcher 丢失

`RefreshProviders` 刷新后重置 `m_providers_dirty = false`，并**保留旧节点的故障状态**（故障节点不会因为刷新而"复活"）。

### 整体流程图

```
ZkClient::Start()
  │  zookeeper_init → global_watcher 回调 → condition_variable 通知
  │
客户端发现服务：
  GetChildrenWithWatcher(path, ChildWatcher, this)
  │  同时做两件事：返回当前子节点列表 + 注册 watcher
  │
ZK 节点变化：
  ChildWatcher() → m_providers_dirty = true
  │  下次 CallMethod → ConnectToAvailableProvider → RefreshProviders
```

## 11. Muduo Reactor 网络模型

### 核心概念

Muduo 是基于 Reactor 模式的 C++ 网络库：
- **非阻塞 IO + epoll**（IO 多路复用）
- **事件循环**（EventLoop）：主线程 accept，IO 线程处理读写
- **连接管理**：自动管理 TCP 连接生命周期

### 事件驱动流程

```
epoll_wait()
  ├─ 新连接 → OnConnection()
  └─ 可读事件 → handleRead()
                   └─ messageCallback_(conn, buffer, timestamp)
                        └─ KrpcProvider::OnMessage()
```

### 如何解耦网络 IO 与 RPC 方法调用？

1. **回调绑定**：`std::bind` 将成员函数和 `this` 绑定成普通函数对象，muduo 不需要知道 KrpcProvider 的存在
2. **自定义协议**：网络层只处理字节流，业务层只处理 Protobuf 对象
3. **done 回调**：`NewCallback` 将 `SendRpcResponse` 包装为 `Closure`，业务代码调用 `done->Run()` 才真正回包

### 为什么客户端用 raw socket 而不是 muduo？

客户端只需要一条 TCP 连接，不需要 Reactor 的多路复用能力。直接 `socket()` + `connect()` + `send()` + `recv()` 更简单，没有额外依赖。

## 12. 自定义通信协议设计

### 消息格式

```
请求：TotalLen(4B) + HeaderLen(4B) + RpcHeader(Protobuf) + Args(Protobuf)
  RpcHeader { service_name, method_name, args_size }

响应：TotalLen(4B) + ResponseData(Protobuf)
```

### 设计思路

- `TotalLen` 在最前面 → 切分包边界
- `HeaderLen` 分离 → 先解析可变长 Header，再读 Body
- 所有整数网络字节序 → 跨平台
- Protobuf 序列化 → 紧凑、高效、兼容性好

### 打包代码（`CallMethod` 中）

```cpp
uint32_t header_size = rpc_header_str.size();
uint32_t total_len = 4 + header_size + args_str.size();
uint32_t net_total_len = htonl(total_len);
uint32_t net_header_len = htonl(header_size);

send_rpc_str.append((char*)&net_total_len, 4);
send_rpc_str.append((char*)&net_header_len, 4);
send_rpc_str.append(rpc_header_str);
send_rpc_str.append(args_str);
```

## 13. 服务注册与发现的高可用性保障

ZooKeeper 本身满足**强一致性**（CP），使用 ZAB 协议保证各节点数据一致。

**故障处理：**
- ZK 集群故障节点不超过半数时，集群仍正常服务
- 服务实例为**临时节点**（`ZOO_EPHEMERAL`），断开后自动删除
- 双重刷新（Watcher + 定时）及时发现变化

**`RefreshProviders` 的细节：**
- 清空旧列表前，把旧节点的故障状态（alive + last_failed）**继承**到新列表的同 ip:port 节点——避免故障节点因刷新"复活"
- 使用 `std::unique_lock<std::mutex>` 保护 ZK 操作

**服务注册流程（`KrpcProvider::Run`）：**

```cpp
// 1. 创建 service_name 永久节点
zkclient.Create(service_path.c_str(), nullptr, 0);
// 2. 创建 method_name 永久节点
zkclient.Create(method_path.c_str(), nullptr, 0);
// 3. 创建 ip:port 临时节点（存 "ip:port"）
zkclient.Create(instance_path.c_str(), data, len, ZOO_EPHEMERAL);
```

## 14. Glog 线程安全详解

### 线程安全设计

**1. 内部锁机制：**
Glog 使用互斥锁保护共享资源（日志文件写入、内存缓冲区）。每次只有一个线程写入日志文件，避免数据竞争。

**2. ThreadLocal 缓冲区：**
每个线程有独立的日志缓冲区。多线程同时写日志时，先写线程自己的缓冲区，减少全局锁竞争。缓冲区满后才刷新到磁盘。

**3. 延迟写入：**
只在缓冲区满、程序退出等时机才真实写磁盘，避免频繁 IO。

### 异步写入特性

虽不是严格意义上的异步日志库（如 spdlog），但设计有类似特性：
- 用户线程只写内存缓冲区，磁盘写入由后台线程完成
- `Fatal` 级别立即刷新所有缓冲，确保崩溃信息不丢失

### 项目中的封装

```cpp
class KrpcLogger {
public:
    explicit KrpcLogger(const char *argv0) {
        google::InitGoogleLogging(argv0);
        FLAGS_colorlogtostderr = true;
        FLAGS_logtostderr = true;
    }
    ~KrpcLogger() { google::ShutdownGoogleLogging(); }
    static void Info(const std::string &message)    { LOG(INFO) << message; }
    static void Warning(const std::string &message) { LOG(WARNING) << message; }
    static void ERROR(const std::string &message)   { LOG(ERROR) << message; }
    static void Fatal(const std::string &message)   { LOG(FATAL) << message; }
};
```

## 15. 服务端核心机制

### 服务注册（NotifyService）

```cpp
void KrpcProvider::NotifyService(google::protobuf::Service *service) {
    const ServiceDescriptor *psd = service->GetDescriptor();
    std::string service_name = psd->name();
    for (int i = 0; i < psd->method_count(); ++i) {
        const MethodDescriptor *pmd = psd->method(i);
        service_info.method_map.emplace(pmd->name(), pmd);
    }
    service_info.service = service;
    service_map.emplace(service_name, service_info);
}
```

只存到内存的 `service_map`，不涉及网络。真正的网络启动在 `Run()`。

### 启动服务（Run）

```cpp
void KrpcProvider::Run() {
    // 1. 读配置 → 2. 创建 muduo TcpServer 绑定回调
    // 3. 连接 ZK → 注册所有服务的永久节点 + 临时实例节点
    // 4. server->start() → event_loop.loop() 进入事件循环
}
```

### 消息处理（OnMessage → 反射分发）

```
OnMessage 解出 service_name & method_name
  │
  ├─ service_map 查找对应服务和方法
  ├─ 反射创建 request → ParseFromString
  ├─ 反射创建 response
  ├─ NewCallback 包装 SendRpcResponse → done
  │
  └─ service->CallMethod(method, controller, request, response, done)
       │
       ▼
     UserServiceRpc::CallMethod()        ← Protobuf 生成
       │  switch(method->index())
       │    case 0: Login(controller, request, response, done)
       ▼
     UserService::Login(...)             ← 你重写的业务逻辑
       │  填充 response
       └─ done->Run()
            │
            ▼
          SendRpcResponse(conn, response) ← 框架回包
```

两层虚函数调用完成框架层 → 业务层的跳转，框架不需要知道具体服务。

### 响应发送（SendRpcResponse）

```cpp
void KrpcProvider::SendRpcResponse(const TcpConnectionPtr &conn, Message *response) {
    std::string response_str;
    response->SerializeToString(&response_str);
    uint32_t len = response_str.size();
    uint32_t net_len = htonl(len);
    send_buf.resize(4 + len);
    memcpy(&send_buf[0], &net_len, 4);
    memcpy(&send_buf[4], response_str.data(), len);
    conn->send(send_buf);
}
```

## 16. 连接池与负载均衡（最新代码）

### 连接池实现

```cpp
// Key = "ip:port"，Value = socket fd
std::unordered_map<std::string, int> m_conn_pool;

// 四个操作：
std::string PoolKey(ip, port);       // "192.168.0.1:8000"
int  GetPooledFd(ip, port);          // 池中有 → 复用；没有 → 返回 -1
void AddToPool(ip, port, fd);        // 新建连接后加入
void RemoveFromPool(ip, port);       // 断开并移除
```

### 两种场景的差异化策略

| 场景 | 调用成功后 | 原因 |
|------|-----------|------|
| 单实例（`m_providers.size() == 1`） | 连接留在池中，复用 | 避免频繁三次握手 |
| 多实例（`m_providers.size() > 1`） | 从池中移除 | 下次调用重新随机选节点，实现负载均衡 |

### ConnectToAvailableProvider 的完整流程

```
阶段一：判断是否需要刷新
  m_providers 为空 || dirty 标记为 true || 距上次刷新 > 10秒
  → RefreshProviders() 从 ZK 拉最新列表

阶段二：第一轮尝试
  while (SelectProvider → 随机选存活节点) {
      if (GetPooledFd 池中有该节点) → 复用，返回
      if (newConnect 新建连接成功) → AddToPool，返回
      MarkProviderFailed → 换下一个节点
  }

阶段三：第二轮兜底
  再次 RefreshProviders() → 重新随机尝试
  // 都失败 → 整个 CallMethod 失败
```

## 17. 这个框架还缺少什么？（面试加分题）

- **超时控制**：`recv_exact` 没有超时设置，服务端卡住客户端会阻塞
- **异步调用**：`CallMethod` 是同步阻塞的，不支持 Future/Callback 异步模式
- **鉴权/限流/监控**：生产环境必备的基础设施
- **服务权重**：所有实例平等，无法按权重分配流量
- **TLS 加密**：内网场景够用，外网需要加密传输
