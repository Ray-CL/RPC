# Krpc RPC 框架面试问题详解

## 一、项目概览

**Q: 一句话介绍这个项目？**

基于 C++11 的轻量级 RPC 框架，使用 Muduo 网络库实现 Reactor 模式，Protobuf 做序列化，ZooKeeper 做服务注册与发现。

**Q: 核心分层架构？**

```
业务层：  UserService::Login()          stub.Login()
            ↑                              ↓
框架层：  KrpcProvider                 KrpcChannel
            ↑                              ↓
网络层：  Muduo TcpServer              raw TCP socket
            ↑                              ↓
注册中心： ZkClient ←──── ZooKeeper ────→ ZkClient
```

## 二、协议设计

**Q: 自定义协议格式是什么？为什么这么设计？**

请求包：
```
[4B TotalLen] [4B HeaderLen] [Protobuf RpcHeader] [Protobuf Args]
```

响应包：
```
[4B TotalLen] [Protobuf ResponseData]
```

RpcHeader 包含：
```protobuf
message RpcHeader {
    bytes service_name = 1;  // 服务名，如 "Kuser.UserServiceRpc"
    bytes method_name  = 2;  // 方法名，如 "Login"
    uint32 args_size   = 3;  // 参数长度
}
```

**设计原因**：
- `TotalLen` 在最前面是为了**解决 TCP 粘包/拆包**——用总长度界定每个包的边界
- `HeaderLen + Header` 分离是为了**解耦**——先知道头部多大，再解析头部，知道 body 多大再读 body
- 所有整数字段用**网络字节序**（`htonl`/`ntohl`），保证跨平台

**Q: 粘包/拆包是怎么处理的？**

TCP 是流式协议，没有消息边界。可能一次收到半个包（拆包），也可能一次收到多个包（粘包）。

客户端用 `recv_exact` 循环接收，不收够不返回：
```cpp
while (total_read < size) {
    ret = recv(fd, buf + total_read, size - total_read, 0);
    if (ret <= 0) return ret;
    total_read += ret;
}
```

服务端用 while 循环 + peek 预读：
```cpp
while (buffer->readableBytes() >= 4) {           // 至少能读 4 字节
    total_len = peek 4 字节;                       // 预读，不消费
    if (buffer->readableBytes() < 4 + total_len) { // 完整包没到齐
        break;                                     // 等下次 OnMessage 再拼
    }
    // 完整包到了，开始解包...
}
```

## 三、服务注册与发现（ZooKeeper）

**Q: ZooKeeper 上的节点结构是什么？**

```
/Kuser.UserServiceRpc              ← 永久节点（服务名）
  ├── /Login                       ← 永久节点（方法名）
  │   ├── /127.0.0.1:8000          ← 临时节点，存 "127.0.0.1:8000"
  │   └── /127.0.0.1:8001          ← 临时节点，存 "127.0.0.1:8001"
  └── /Register                    ← 永久节点
      └── /127.0.0.1:8000          ← 临时节点
```

**Q: 为什么方法节点是永久的，实例节点是临时的？**

- **永久节点**（服务名、方法名）：和方法名存在一起，即使所有实例下线，方法名还在，新实例上线直接添加到下面即可
- **临时节点**（`ip:port`）：服务实例断开连接后，ZooKeeper 自动删除该节点，客户端刷新列表时自然就拿不到已下线的实例了。不需要服务端主动注销

**Q: 客户端怎么发现服务？**

```
1. ZkClient::GetChildren("/Kuser.UserServiceRpc/Login")
   → 返回 ["127.0.0.1:8000", "127.0.0.1:8001"]

2. 逐个 ZkClient::GetData(".../127.0.0.1:8000")
   → 返回 "127.0.0.1:8000"

3. 解析出 ip 和 port，存入 m_providers 列表

4. 每 10 秒刷新一次列表
```

**Q: 为什么节点名叫 `ip:port`，节点数据也存 `ip:port`？**

ZooKeeper C API 的 `zoo_get_children` 只返回子节点**名称**，不直接给数据。虽然可以直接从节点名解析出 ip:port，但为了通用性（节点名可能不是 ip:port 格式），代码选择再调一次 `GetData` 取数据。实际部署中存一样的值也是常见做法。

## 四、序列化（Protobuf）

**Q: 为什么选 Protobuf？**

- **跨语言**：支持 C++/Java/Python/Go 等多语言
- **高效**：二进制序列化，比 JSON/XML 体积小、速度快
- **代码生成**：通过 `.proto` 文件自动生成 Service/Stub 代码，只需重写虚函数即可
- **反射**：`ServiceDescriptor`/`MethodDescriptor` 让服务端可以**动态创建请求/响应对象**，不需要知道具体类型

**Q: 服务端用到的反射是怎么做的？**

```cpp
// 不需要写 new LoginRequest()——通用代码不依赖具体类型
Message* request  = service->GetRequestPrototype(method).New();  // 动态创建请求
Message* response = service->GetResponsePrototype(method).New(); // 动态创建响应
request->ParseFromString(args_str);                              // 反序列化
```

`GetRequestPrototype` 返回这个方法的输入类型**原型**，`.New()` 克隆出一个空对象。这样 `OnMessage` 就是完全通用的，加新服务不用改框架代码。

## 五、网络模型（Muduo/Reactor）

**Q: muduo 的 Reactor 模式在这个项目里怎么工作的？**

```
epoll_wait() 监听
  │
  ├─ 新连接事件 → OnConnection()
  │
  └─ 可读事件 → TcpConnection::handleRead()
                   │  从 socket 读数据到 Buffer
                   └─→ messageCallback_(conn, buffer, timestamp)
                         │
                         └─→ KrpcProvider::OnMessage(conn, buffer, timestamp)
```

`setThreadNum(4)` 表示 4 个 IO 线程处理读写。主线程负责 accept，4 个 worker 线程轮询处理已建立连接的 IO 事件。

**Q: 绑定回调的时候 std::bind 做了什么？**

```cpp
server->setMessageCallback(
    std::bind(&KrpcProvider::OnMessage, this, _1, _2, _3)
);
```

`std::bind` 把成员函数 `OnMessage` 和 `this` 指针绑定，生成一个**普通函数对象**。muduo 内部以 `function<void(conn, buffer, timestamp)>` 的形式存储，收到数据时直接调用。这样 muduo（网络层）不需要知道 KrpcProvider（业务层）的存在。

## 六、客户端核心机制

### 6.1 CallMethod 完整流程

```
stub.Login(&controller, &request, &response, nullptr)
  │
  ├─ UserServiceRpc_Stub::Login()
  │     └─ channel_->CallMethod(descriptor()->method(0), ...)
  │
  └─ KrpcChannel::CallMethod()
        ├─ 1. 服务发现：ZK 查 ip:port → TCP connect
        ├─ 2. 序列化请求：request → args_str
        ├─ 3. 构建 RpcHeader：service_name, method_name, args_size
        ├─ 4. 打包：[total_len][header_len][header][args] → send()
        ├─ 5. 收响应：recv_exact(4) → recv_exact(response_len)
        └─ 6. 反序列化：response->ParseFromArray(buf, len)
```

### 6.2 服务发现与重试机制（ConnectToAvailableProvider）

```
阶段一：判断是否需要刷新
  m_providers 为空 OR 距离上次刷新 > 10 秒 → RefreshProviders()

阶段二：第一轮尝试
  while (SelectProvider(ip, port)) {   // 逐个选存活节点
      if (newConnect(ip, port) 成功) → return true;
      MarkProviderFailed(ip, port);     // 标记失败，换下一个
  }

阶段三：第二轮兜底
  RefreshProviders() → while (SelectProvider(ip, port)) → newConnect()
  // 再失败 → return false → 上层报 "no available service provider"
```

### 6.3 负载均衡是怎么做的？

在所有**存活节点**（`alive=true` 或隔离期满的）中**随机选一个**：

```cpp
// 筛出存活节点索引
for (size_t i = 0; i < m_providers.size(); ++i) {
    if (ProviderAlive(m_providers[i])) {
        alive_indices.push_back(i);
    }
}
// 随机选
size_t chosen = alive_indices[random() % alive_indices.size()];
```

简单有效，不需要维护计数器。如果服务实例是同构的（处理能力相同），随机和轮询效果接近。

### 6.4 故障隔离怎么实现？

**标记失败**（`MarkProviderFailed`）：
- 连接失败、发送失败、接收失败、反序列化失败时调用
- 设置 `alive = false`，记录 `last_failed = now`

**判断存活**（`ProviderAlive`）：
- `alive == true` → 直接可用
- `alive == false` 且 距今 < 3 秒 → 不可用（隔离中）
- `alive == false` 且 距今 >= 3 秒 → 重新可用（自动恢复）

**为什么是 3 秒自动恢复而不是永久拉黑？**
- 可能是临时网络抖动，短暂恢复后重新尝试
- 如果真的永久挂了，ZK 临时节点会自动删除，下次 `RefreshProviders` 就拉不到了

### 6.5 连接管理策略

```cpp
// CallMethod 成功后：
if (m_providers.size() > 1) {
    close(m_clientfd);
    m_clientfd = -1;   // 多实例：断开，下次随机连另一个（负载均衡）
}
// 单实例：保持连接，复用（性能最优）
```

| 场景 | 策略 | 原因 |
|------|------|------|
| 单实例 | 连接复用 | 避免频繁三次握手 |
| 多实例 | 每次断开重连 | 随机选不同节点，实现负载均衡 |

这是一种**用性能换负载均衡**的简单方案。生产环境可以用**连接池**优化：连过的 fd 缓存起来，随机选到相同节点时直接复用。

### 6.6 CallMethod 的重试循环

```cpp
for (int attempt = 0; attempt < 2; ++attempt) {
    // 没有可用连接 → 去连
    if (m_clientfd == -1) {
        ConnectToAvailableProvider(...);  // 内部会换节点
    }
    // 发送 → 收响应 → 反序列化
    // 任何一步失败 → MarkProviderFailed → close → m_clientfd = -1 → continue 重试
}
```

最多 2 次尝试。第一次失败后 `m_clientfd` 被重置为 -1，第二次循环会重新 `ConnectToAvailableProvider`，但**不会选到刚刚失败的节点**（它在 3 秒隔离期内）。

## 七、服务端核心机制

### 7.1 服务注册（NotifyService）

```cpp
provider.NotifyService(new UserService());
```

- 反射获取服务名和方法列表
- 存入 `service_map`：

```
service_map = {
    "Kuser.UserServiceRpc" → {
        service: UserService*,
        method_map: {
            "Login"    → MethodDescriptor(Login),
            "Register" → MethodDescriptor(Register)
        }
    }
}
```

**只是存到内存里**，不涉及网络操作。真正的网络启动在 `Run()` 里。

### 7.2 启动服务（Run）

```cpp
void KrpcProvider::Run() {
    // 1. 读配置：rpcserverip, rpcserverport
    // 2. 创建 muduo TcpServer，绑定 OnConnection/OnMessage 回调
    // 3. 向 ZK 注册所有服务（永久节点 + 临时实例节点）
    // 4. server->start() → event_loop.loop() → 阻塞，进入事件循环
}
```

`event_loop.loop()` 之后的所有逻辑都由**回调驱动**，不再有顺序执行的代码。

### 7.3 消息处理（OnMessage）

```
muduo 收到数据
  │
  ▼
OnMessage(conn, buffer, timestamp)
  │
  ├─ while 循环解包（粘包处理）
  │    读 total_len → 读 header_len → 读 RpcHeader → 读 args
  │    得到 service_name, method_name, args_str
  │
  ├─ service_map 查找对应服务和方法
  │
  ├─ 反射创建 request（从 args_str 反序列化）
  ├─ 反射创建 response（空对象）
  ├─ 包装 done → SendRpcResponse(conn, response)
  │
  └─ service->CallMethod(method, nullptr, request, response, done)
```

### 7.4 回调机制（done->Run()）

**为什么需要 done？** 业务代码和网络代码分离。UserService::Login 只管填 response，不用管怎么发回去。

```cpp
// OnMessage 里：
Closure* done = NewCallback<KrpcProvider, const TcpConnectionPtr&, Message*>(
    this, &KrpcProvider::SendRpcResponse, conn, response
);

// NewCallback 内部 new 了一个 MethodClosure2<KrpcProvider, TcpConnectionPtr&, Message*>：
//   object_ = this (KrpcProvider*)
//   method_ = &KrpcProvider::SendRpcResponse
//   arg1_   = conn
//   arg2_   = response
//
//   void Run() override {
//       (object_->*method_)(arg1_, arg2_);  // = this->SendRpcResponse(conn, response)
//   }
```

NewCallback **只创建对象，不执行**。只有业务代码调用 `done->Run()` 时才真正触发 `SendRpcResponse`。

### 7.5 反射分发链

```
service->CallMethod(method, nullptr, request, response, done)
  │  service 实际指向 UserService 对象
  │  CallMethod 是虚函数
  ▼
UserServiceRpc::CallMethod()          ← protobuf 生成
  │  switch(method->index())
  │    case 0: Login(controller, request, response, done)
  ▼
UserService::Login(...)               ← 你重写的业务逻辑
  │  从 request 读参数
  │  执行业务逻辑
  │  填充 response
  └─ done->Run()
       │
       ▼
     SendRpcResponse(conn, response)  ← 框架回包
```

两次虚函数调用完成**通用框架层 → 业务层**的跳转。框架不需要知道具体有哪些服务。

## 八、Design Decisions 面试追问

**Q: 为什么客户端用 raw socket 而不是 muduo？**

客户端只需要一条 TCP 连接，不需要 Reactor 模式的多路复用能力。直接用 `socket()` + `connect()` + `send()` + `recv()` 更简单，没有额外依赖。

**Q: 为什么用 Protobuf 的 Service 机制而不是自己定义？**

Protobuf 的 Service 提供了：
1. `ServiceDescriptor` / `MethodDescriptor` 反射——服务端可以动态创建请求/响应
2. `Stub` 代码生成——客户端只需 `stub.Login(...)` 一行调用
3. `RpcChannel` 接口——只需实现 `CallMethod`，框架自动对接

自己定义也能实现，但等于重新发明轮子。

**Q: 粘包问题为什么用 total_len 前置而不是分隔符？**

- 二进制协议用分隔符不可靠（分隔符可能出现在数据中）
- 长度前置是二进制协议的标准做法（HTTP 的 Content-Length、gRPC 等都用）
- 额外开销只有 4 字节

**Q: 如果服务端挂了，客户端要多久才能发现？**

最坏情况：刚刷新完列表（0 秒）+ 10 秒缓存期 + 重试两次 = 约 10 秒。但实际上 ZK 临时节点的会话超时默认是心跳的倍数，服务端进程崩溃后 ZK 需要先感知心跳超时才会删除节点。优化方向：缩短刷新间隔，或加 ZK watcher 主动推送节点变化。

**Q: 这个框架缺少什么？**

- **超时控制**：`recv_exact` 没有超时，服务端卡住客户端会死等
- **连接池**：多实例场景每次断开重连，影响性能
- **异步回调**：`CallMethod` 是同步阻塞的，不支持异步调用
- **鉴权/限流/监控**：生产环境必备
- **服务版本/权重**：无法给不同实例分配不同权重
