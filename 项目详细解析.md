## 1. 整体架构

```
┌──────────────────────────────────────────────────────────────┐
│                       ZooKeeper 集群                         │
│  /UserServiceRpc/Login/192.168.1.10:8000  (临时节点)         │
│  /UserServiceRpc/Login/192.168.1.11:8000  (临时节点)         │
│  /UserServiceRpc/Register/192.168.1.10:8000 (临时节点)       │
└──────┬──────────────────────────────────────┬────────────────┘
       │ 注册 (Create ephemeral)              │ 发现 (GetChildren)
       ▼                                      ▼
┌──────────────────┐                  ┌──────────────────┐
│   KrpcProvider   │                  │   KrpcChannel    │
│  (example/callee)│                  │  (example/caller)│
└────────┬─────────┘                  └────────┬─────────┘
         │                                     │
    ┌────▼────┐                           ┌───▼────┐
    │  Muduo  │                           │  Raw   │
    │TcpServer│                           │ Socket │
    └────┬────┘                           └───┬────┘
         │                                     │
         └─────────── TCP 通信 ────────────────┘
                  自定义二进制协议
```

### 分层职责

| 层级 | 组件 | 职责 |
|------|------|------|
| 业务层 | `example/callee/Kserver.cc` / `example/caller/Kclient.cc` | 定义 protobuf 服务接口，实现业务逻辑 |
| 框架层 | `KrpcProvider` / `KrpcChannel` | 服务发布、连接管理、协议编解码、方法路由 |
| 网络层 | Muduo TcpServer（服务端）/ Raw TCP Socket（客户端） | 异步事件驱动 IO（Reactor 模式） |
| 注册中心 | `ZkClient` | 服务注册（临时节点）、服务发现（子节点列表） |
| 序列化 | Protobuf | 请求/响应消息的序列化与反序列化 |

## 2. 启动流程

### 2.1 服务端启动（example/callee/Kserver.cc）

```
main()
 │
 ├─1. KrpcApplication::Init(argc, argv)
 │   └─ 解析命令行参数 -i <配置文件路径>
 │   └─ Krpcconfig::LoadConfigFile() 逐行解析 key=value 配置
 │
 ├─2. 构造 KrpcProvider provider
 │
 ├─3. provider.NotifyService(new UserService())
 │   └─ 遍历 protobuf ServiceDescriptor，将 服务名→方法名→MethodDescriptor 的映射存入 service_map
 │
 └─4. provider.Run()
     ├─ 从配置读取 rpcserverip / rpcserverport
     ├─ 创建 muduo::TcpServer，绑定 OnConnection / OnMessage 回调
     ├─ server->setThreadNum(4) → 4 个 IO 线程
     ├─ 连接 ZooKeeper，将每个服务方法的 ip:port 作为临时节点注册
     └─ server->start() + event_loop.loop() → 进入事件循环，阻塞等待请求
```

### 2.2 客户端启动（example/caller/Kclient.cc）

```
main()
 │
 ├─1. KrpcApplication::Init(argc, argv)
 │   └─ 同上，解析命令行并加载配置
 │
 ├─2. KrpcLogger logger("MyRPC") → 初始化 glog
 │
 └─3. 启动多线程压测
     └─ 每个线程：
         ├─ 创建 UserServiceRpc_Stub stub(new KrpcChannel(false))
         │   └─ KrpcChannel(false) 不立即连接，延迟到 CallMethod 时按需连接
         ├─ 构造 LoginRequest，设置 name/pwd
         └─ 循环 N 次调用 stub.Login(&controller, &request, &response, nullptr)
```

## 3. 服务注册流程

发生在 `KrpcProvider::Run()` 中，对应源码 `src/Krpcprovider.cc:65-84`。

```
for 每个已注册的 ServiceInfo:
    service_path = "/UserServiceRpc"
    │
    │ zkclient.Create(service_path, ...)        ← 永久节点
    │
    for 每个 MethodDescriptor:
        method_path = "/UserServiceRpc/Login"
        │
        │ zkclient.Create(method_path, ...)     ← 永久节点
        │
        instance_path = "/UserServiceRpc/Login/192.168.1.10:8000"
        │
        zkclient.Create(instance_path, ..., ZOO_EPHEMERAL)  ← 临时节点
        └─ 节点数据 = "192.168.1.10:8000"
```

**节点类型设计意图：**
- **永久节点**（服务名、方法名）：即使所有服务实例都下线，目录结构保留，新实例可以直接在对应方法下注册。
- **临时节点**（ip:port 实例）：服务进程断开 ZooKeeper 连接后，临时节点自动删除，客户端刷新列表时即可感知实例下线。

## 4. RPC 调用完整流程

以下是一次 `stub.Login(...)` 调用的完整数据流。

### 4.1 客户端：发送请求（KrpcChannel::CallMethod）

```
CallMethod(method, controller, request, response, done)
 │
 ├─1. 提取 service_name / method_name
 │
 ├─2. EnsureZkClient() → 延迟创建 ZkClient 并连接 ZooKeeper
 │
 ├─3. ConnectToAvailableProvider()
 │   ├─ 每 10 秒刷新一次提供者列表（从 ZooKeeper 拉取子节点）
 │   ├─ SelectProvider() → 随机选择一个存活节点
 │   │   └─ 过滤掉被标记为 failed 且在 3 秒隔离期内的节点
 │   └─ newConnect(ip, port) → raw socket connect()
 │       └─ 成功则将 ip:port 记入 m_ip/m_port
 │
 ├─4. 序列化请求：
 │   request → SerializeToString() → args_str (protobuf 二进制)
 │
 ├─5. 构建 RpcHeader：
 │   RpcHeader { service_name, method_name, args_size }
 │   → SerializeToString() → rpc_header_str
 │
 ├─6. 拼装数据包并发送：
 │   ┌──────────────┬──────────────┬──────────────────┬─────────────┐
 │   │ 4B TotalLen  │ 4B HeaderLen │ RpcHeader (pb)   │ Args (pb)   │
 │   │  (网络序)    │  (网络序)    │                  │             │
 │   └──────────────┴──────────────┴──────────────────┴─────────────┘
 │   send(m_clientfd, ...)
 │
 ├─7. 接收响应：
 │   recv_exact(4) → 读取 response_len (网络序 → 主机序)
 │   recv_exact(response_len) → 读取响应体
 │
 └─8. response->ParseFromArray() → 反序列化得到结果
     └─ 如果有多实例（m_providers.size() > 1），关闭连接以便下次重新选择
```

### 4.2 服务端：接收并处理请求（KrpcProvider::OnMessage）

```
OnMessage(conn, buffer, timestamp)  // muduo 回调
 │
 └─ while (buffer->readableBytes() >= 4)  ← 循环处理粘包
     │
     ├─1. peek() 读取 total_len（4 字节），不移动读指针
     │
     ├─2. 检查包完整性：
     │     if (readableBytes < 4 + total_len) → break;  // 拆包，等待更多数据
     │
     ├─3. retrieve(4) → 消费 total_len 字段
     │
     ├─4. 读取 header_len（4 字节网络序）→ 转主机序
     │
     ├─5. 读取 RpcHeader 并反序列化 → 得到 service_name, method_name
     │
     ├─6. 计算 args_size → 读取 Args 并反序列化 → 得到 request 对象
     │
     ├─7. 查找路由：
     │     service_map.find(service_name) → method_map.find(method_name)
     │     → 得到 Service* 和 MethodDescriptor*
     │
     ├─8. 创建 response 对象，构造 Closure 回调：
     │     NewCallback(this, &KrpcProvider::SendRpcResponse, conn, response)
     │     └─ 当业务方法调用 done->Run() 时，自动触发 SendRpcResponse
     │
     └─9. service->CallMethod(method, nullptr, request, response, done)
          └─ 多态调用 → 进入业务层重写的 Login() 方法
```

### 4.3 服务端：发送响应（KrpcProvider::SendRpcResponse）

```
SendRpcResponse(conn, response)
 │
 ├─ response->SerializeToString() → 序列化响应数据
 │
 └─ 打包并发送：
     ┌──────────────┬─────────────────────┐
     │ 4B TotalLen  │ ResponseData (pb)   │
     │  (网络序)    │                     │
     └──────────────┴─────────────────────┘
     conn->send(send_buf)
```

### 4.4 业务层回调链（服务端业务方法）

```
UserService::Login(controller, request, response, done)
 │
 ├─ 从 request 提取参数（name, pwd）
 ├─ 调用本地 Login(name, pwd) 执行业务逻辑
 ├─ 填充 response（errcode, success）
 └─ done->Run()
     └─ → KrpcProvider::SendRpcResponse(conn, response)  ← 由 protobuf Closure 回调
         └─ 序列化 response 并发送给客户端
```

## 5. 自定义协议与粘包处理

### 5.1 协议格式

**请求包：**

```
┌────────────────┬──────────────┬──────────────────┬────────────┐
│ TotalLen (4B)  │ HeaderLen(4B)│ RpcHeader (pb)   │ Args (pb)  │
│ 大端网络字节序 │ 大端网络字节序│ 可变长           │ 可变长     │
└────────────────┴──────────────┴──────────────────┴────────────┘
│                │               │                  │            │
│ total_len = 4 + header_len + args_size            │            │
│                                                   │            │
│ header_len = RpcHeader 序列化后长度               │            │
└───────────────────────────────────────────────────────────────┘
```

**响应包（更简洁）：**

```
┌────────────────┬────────────────────┐
│ TotalLen (4B)  │ ResponseData (pb)  │
│ 大端网络字节序 │ 可变长             │
└────────────────┴────────────────────┘
```

### 5.2 RpcHeader 定义（src/Krpcheader.proto）

```protobuf
message RpcHeader {
    bytes service_name = 1;  // 服务名，如 "UserServiceRpc"
    bytes method_name  = 2;  // 方法名，如 "Login"
    uint32 args_size   = 3;  // 参数体的字节数
}
```

### 5.3 粘包/拆包处理

TCP 是流式协议，多次 write 的数据可能在接收端合并（粘包），也可能被内核拆分成多段到达（拆包）。

**服务端处理策略**（`OnMessage` 中的 while 循环）：

1. 用 `peek()` 预读前 4 字节（TotalLen），不移动缓冲区读指针
2. 检查 `readableBytes >= 4 + total_len`：如果数据不够一个完整包，`break` 退出循环，等待下次 `OnMessage` 触发
3. 数据完整时，用 `retrieve()` 逐步消费各字段
4. `while` 循环处理下一个包（粘包场景）

**客户端处理策略**（`recv_exact` 函数）：

```cpp
ssize_t recv_exact(int fd, char* buf, size_t size) {
    size_t total_read = 0;
    while (total_read < size) {
        ssize_t ret = recv(fd, buf + total_read, size - total_read, 0);
        if (ret == 0) return 0;       // 对端关闭
        if (ret == -1) {
            if (errno == EINTR) continue;  // 被信号中断，继续读
            return -1;
        }
        total_read += ret;
    }
    return total_read;
}
```

循环读取直到凑满指定字节数，解决了客户端接收响应时的拆包问题。

## 6. ZooKeeper 服务发现机制

### 6.1 连接建立（ZkClient::Start）

```cpp
void ZkClient::Start() {
    // 1. 读取 zookeeperip:zookeeperport 配置
    // 2. 调用 zookeeper_init() 异步建立连接
    // 3. 注册 global_watcher 回调
    // 4. 用 condition_variable 阻塞等待 ZOO_CONNECTED_STATE 事件
    // 5. 连接成功后返回
}
```

`zookeeper_mt`（多线程版）内部维护三个线程：
- API 调用线程：执行 zoo_get/zoo_exists 等同步 API
- 网络 IO 线程：基于 poll 的事件循环
- watcher 回调线程：执行用户注册的回调函数

### 6.2 节点创建（ZkClient::Create）

```cpp
void ZkClient::Create(const char* path, const char* data, int datalen, int state) {
    // 1. zoo_exists() 检查节点是否已存在
    // 2. 如果 ZNONODE → zoo_create() 创建节点
    //    state=0 → 永久节点
    //    state=ZOO_EPHEMERAL → 临时节点（会话断开后自动删除）
}
```

### 6.3 客户端服务发现

客户端通过 `RefreshProviders()`（`src/Krpcchannel.cc:194`）从 ZooKeeper 获取可用服务列表：

```
method_path = "/UserServiceRpc/Login"

zoo_get_children(method_path) → ["192.168.1.10:8000", "192.168.1.11:8000"]

for each child:
    zoo_get_data("/UserServiceRpc/Login/192.168.1.10:8000")
    → "192.168.1.10:8000"  (即时节点注册时写入的数据)
    → 解析出 ip 和 port
    → 构建 ProviderInfo 列表
```

**刷新策略**：
- 首次调用或距上次刷新超过 10 秒时，重新从 ZooKeeper 拉取列表
- 调用 `ZkClient::GetChildren/GetData` 时加全局互斥锁 `g_data_mutx`，因为 ZooKeeper C 客户端的 watcher 回调可能并发访问

## 7. 负载均衡与故障隔离

### 7.1 随机负载均衡（SelectProvider）

```
1. 遍历 m_providers，筛选 alive_indices（存活的提供者索引）
2. 用 thread_local std::mt19937 随机数生成器 + uniform_int_distribution
   从 alive_indices 中随机选取一个节点
3. m_provider_index = (chosen + 1) % m_providers.size()  ← 预留的轮询状态
```

### 7.2 故障隔离（3 秒冷却期）

当连接或通信失败时：

```
MarkProviderFailed(ip, port):
    provider.alive = false
    provider.last_failed = now()
```

下次选择提供者时的判断逻辑：

```
ProviderAlive(provider):
    if provider.alive → true        // 正常可用
    else:
        if now - last_failed >= 3s → true   // 冷却期满，重新尝试
        else → false                        // 仍在隔离期
```

**设计意义**：避免在故障节点上反复重试失败，同时 3 秒后自动重新探测，无需手动恢复。

### 7.3 连接重试流程（ConnectToAvailableProvider）

```
1. 如果提供者列表为空或超过 10 秒 → RefreshProviders() 刷新列表
2. 循环遍历存活节点：
   ├─ SelectProvider() 随机选一个
   ├─ newConnect(ip, port) 尝试连接
   │   └─ 成功 → 返回 true
   └─ 失败 → MarkProviderFailed() 标记故障，继续尝试下一个
3. 第一轮全部失败 → 再刷新一次列表（防止旧列表全部被标记故障）
4. 第二轮仍失败 → 返回 false
```

### 7.4 CallMethod 中的外层重试

外层有 `for (int attempt = 0; attempt < 2; ++attempt)` 循环，最多两轮尝试。当 `send()`、`recv()` 或反序列化失败时，标记当前提供者、关闭连接、`continue` 进入下一轮，自动选择另一个提供者重试。

## 8. 源码逐文件解析

### 8.1 KrpcApplication（框架入口，单例模式）

**头文件**：`src/include/Krpcapplication.h`
**实现**：`src/Krpcapplication.cc`

### 8.4 KrpcProvider（服务端核心）

**头文件**：`src/include/Krpcprovider.h`
**实现**：`src/Krpcprovider.cc`

**核心数据结构**：
```cpp
struct ServiceInfo {
    google::protobuf::Service* service;          // protobuf 生成的服务对象
    std::unordered_map<std::string, const MethodDescriptor*> method_map;  // 方法名→描述符
};
std::unordered_map<std::string, ServiceInfo> service_map;  // 服务名→ServiceInfo
```

**方法调用链**：

```
NotifyService(service)
    └─ service->GetDescriptor() 获取 ServiceDescriptor
        └─ 遍历 method_count，提取每个方法的 MethodDescriptor
            └─ 存入 service_map

Run()
    ├─ 读取配置 ip:port
    ├─ 创建 muduo::TcpServer，绑定回调
    ├─ 连接 ZooKeeper，注册所有服务的临时节点
    └─ server->start() + event_loop.loop() → 开始监听

OnConnection(conn)
    └─ 连接关闭时调用 conn->shutdown()

OnMessage(conn, buffer, timestamp)
    └─ while 循环解包 → 查找路由 → 调用业务方法

SendRpcResponse(conn, response)
    └─ 序列化 response → 打包 [4B总长 + 数据] → conn->send()
```

### 8.5 KrpcChannel（客户端核心）

**头文件**：`src/include/Krpcchannel.h`
**实现**：`src/Krpcchannel.cc`

**核心数据结构**：
```cpp
struct ProviderInfo {
    std::string ip;
    uint16_t port;
    bool alive = true;
    std::chrono::steady_clock::time_point last_failed;  // 故障时间戳
};
std::vector<ProviderInfo> m_providers;           // 提供者列表
std::unique_ptr<ZkClient> m_zkclient;            // ZooKeeper 客户端（延迟初始化）
std::chrono::steady_clock::time_point m_last_refresh_time;  // 上次刷新时间
```

**核心方法**：

| 方法 | 功能 |
|------|------|
| `CallMethod()` | RPC 调用的入口，负责完整的一次调用流程（连接→序列化→发送→接收→反序列化） |
| `newConnect(ip, port)` | 创建 TCP socket 并连接指定地址 |
| `recv_exact(fd, buf, size)` | 循环读取确保读满指定字节，解决 TCP 拆包 |
| `ConnectToAvailableProvider()` | 从 ZooKeeper 发现服务、选择节点、建立连接 |
| `RefreshProviders()` | 从 ZooKeeper 刷新提供者列表 |
| `SelectProvider()` | 随机负载均衡，跳过故障隔离期内的节点 |
| `MarkProviderFailed()` | 标记节点失败，记录时间戳，触发 3 秒冷却 |
| `ProviderAlive()` | 判断节点是否可用（alive 或超过冷却期） |
| `EnsureZkClient()` | 延迟初始化 ZooKeeper 客户端 |

### 8.6 Krpccontroller（RPC 控制器）

**头文件**：`src/include/Krpccontroller.h`
**实现**：`src/Krpccontroller.cc`

继承自 `google::protobuf::RpcController`，是一个轻量级的状态跟踪器：

```cpp
class Krpccontroller : public google::protobuf::RpcController {
    bool m_failed;          // 调用是否失败
    std::string m_errText;  // 错误描述
};
```

| 方法 | 功能 |
|------|------|
| `Reset()` | 重置状态，复用 controller 对象 |
| `Failed()` | 返回调用是否失败 |
| `SetFailed(reason)` | 标记失败并记录原因 |
| `ErrorText()` | 获取错误描述 |

`StartCancel()`、`IsCanceled()`、`NotifyOnCancel()` 三个方法是 protobuf 接口要求实现的，但本项目未实现取消功能。

### 8.7 ZkClient（ZooKeeper 客户端封装）

**头文件**：`src/include/zookeeperutil.h`
**实现**：`src/zookeeperutil.cc`

**全局变量**（文件作用域）：
```cpp
std::mutex cv_mutex;               // 保护 is_connected
std::condition_variable cv;        // 用于阻塞等待连接完成
bool is_connected = false;         // 连接状态标志
```

**global_watcher 回调**：
```cpp
void global_watcher(zhandle_t *zh, int type, int status, ...) {
    if (type == ZOO_SESSION_EVENT && status == ZOO_CONNECTED_STATE) {
        is_connected = true;
    }
    cv.notify_all();  // 唤醒 Start() 中阻塞的线程
}
```

| 方法 | 功能 |
|------|------|
| `Start()` | 调用 `zookeeper_init()` 异步连接，用 `cv.wait()` 阻塞等待连接成功 |
| `Create(path, data, datalen, state)` | 先 `zoo_exists()` 检查，不存在则 `zoo_create()` |
| `GetData(path)` | 调用 `zoo_get()` 获取节点数据 |
| `GetChildren(path)` | 调用 `zoo_get_children()` 获取子节点名称列表 |

### 8.8 示例：UserService（业务层）

**Proto 定义**：`example/user.proto`

```protobuf
service UserServiceRpc {
    rpc Login(LoginRequest) returns(LoginResponse);
    rpc Register(RegisterRequest) returns(RegisterResponse);
}
```

**服务端实现**（`example/callee/Kserver.cc`）：
- `UserService` 继承 `Kuser::UserServiceRpc`（protoc 生成的基类）
- 重写 `Login()` 虚函数：从 request 提取参数 → 调用本地业务逻辑 → 填充 response → 调用 `done->Run()` 触发响应发送

**客户端调用**（`example/caller/Kclient.cc`）：
- 使用 `Kuser::UserServiceRpc_Stub`（protoc 生成的桩类）
- 创建 `KrpcChannel(false)` 传入 Stub 构造器
- 调用 `stub.Login(&controller, &request, &response, nullptr)`
